## Dev Notes

This story creates the three MVP agent definitions that will be used during the refinement phase (Story 3.1-3.5). Each agent is defined as a SKILL.md file following the framework's declarative agent orchestration pattern.

### Architecture Context

**Agent Loading Model** (from Architecture Decision Document):
- **Agent = Role + Domain Skill + Context**
- **Role**: Defined in `agents/*.md` (this story)
- **Domain Skill**: Project-specific, generated by `/create-project-context` (Story 1.5)
- **Context**: Loaded from `project-root/context/*.md` based on agent's Context Rules

**Model Routing Strategy** (from Architecture):
- Primary model (Opus) for coordination and orchestration
- Lighter models (Sonnet) for sub-agents to reduce token costs 40-60%
- Configured per agent in SKILL.md frontmatter via `model` field

**Table-Based Output Format** (from Architecture):
All three agents use the same table-based refinement perspective format:
- `## [Agent-Name] Perspective` header
- Findings table (columns: #, Finding, Severity, Category)
- Recommendations list
- Proposed Acceptance Criteria checklist

This consistency enables the synthesis skill (Story 3.3) to merge perspectives programmatically.

### SKILL.md Format Specification

**YAML Frontmatter Structure** (must be in this exact order):

```yaml
---
name: architect
display_name: Architect
role: You are an expert software architect focused on system design, risks, and dependencies
active_in:
  - refine-ticket
model: claude-sonnet-4
max_tokens: 2000
---
```

**Frontmatter Field Descriptions:**
- `name`: Short identifier (kebab-case), used in config.yaml active_agents array
- `display_name`: Human-readable name, used in output headers
- `role`: System prompt defining the agent's identity and purpose
- `active_in`: Array of command phases where this agent is activated (e.g., refine-ticket)
- `model`: Model routing - which model to use for this agent (claude-opus-4, claude-sonnet-4, etc.)
- `max_tokens`: Maximum output tokens for this agent's response

**Markdown Body Structure** (must be in this exact order):

1. **Identity**: High-level description of the agent's purpose and expertise
2. **Instructions**: Detailed guidelines for how to analyze stories from this agent's perspective
3. **Output Format**: Specification of the expected output format (table-based refinement perspective)
4. **Context Rules**: Which context files to load and how to use them

### Agent-Specific Requirements

**Architect Agent** (`architect.md`):
- **Purpose**: Identify architectural risks, affected design decisions, and dependencies (FR9)
- **Context Loads**: `story.md` + `context/index.md` + `context/architecture.md` + relevant domain context + `skills/project-architect/SKILL.md`
- **Output Focus**: System design, scalability, maintainability, architectural patterns
- **Severity Levels**: Critical (blocks implementation), Major (significant redesign), Minor (optimization)

**Developer Agent** (`developer.md`):
- **Purpose**: Identify technical dependencies, implementation concerns, and feasibility issues (FR10)
- **Context Loads**: `story.md` + `context/index.md` + relevant domain context + `skills/{ticket-domain}/SKILL.md`
- **Output Focus**: Code complexity, library availability, technical debt, implementation time
- **Severity Levels**: Critical (unfeasible), Major (requires research/spike), Minor (straightforward with notes)

**QA Agent** (`qa.md`):
- **Purpose**: Propose testable acceptance criteria and identify edge cases (FR11)
- **Context Loads**: `story.md` + `context/index.md` + `context/testing.md` + relevant domain context + `skills/testing/SKILL.md`
- **Output Focus**: Test coverage, edge cases, acceptance criteria clarity, testability
- **Severity Levels**: Critical (untestable), Major (ambiguous acceptance criteria), Minor (missing edge cases)

### Table-Based Refinement Perspective Format

All three agents must use this exact output format:

```markdown
## Architect Perspective

### Findings

| # | Finding | Severity | Category |
|---|---------|----------|----------|
| 1 | OAuth2 integration requires secure token storage | Critical | Security |
| 2 | User session state management needs clarification | Major | Architecture |
| 3 | Consider adding refresh token rotation | Minor | Best Practice |

### Recommendations

1. Implement secure token storage using environment variables or a secret manager
2. Clarify user session state management approach (stateless JWT vs. session-based)
3. Add refresh token rotation to enhance security

### Proposed Acceptance Criteria

- [ ] System implements OAuth2 authentication with secure token storage
- [ ] User session state is managed using [specific approach]
- [ ] Refresh token rotation is implemented for enhanced security
```

### Technical Requirements

**File Format Standards:**
- All agent files: Markdown format with YAML frontmatter
- File naming: kebab-case (architect.md, developer.md, qa.md)
- YAML field naming: snake_case (display_name, active_in, max_tokens)
- No trailing whitespace in files
- Unix line endings (LF)

**Frontmatter Validation:**
- All required fields must be present: name, display_name, role, active_in, model, max_tokens
- Fields must be in the exact order specified above
- `active_in` must be a YAML array
- `model` must be a valid model identifier
- `max_tokens` must be a positive integer

**Content Requirements:**
- Each agent's Identity section must clearly differentiate it from the other agents
- Instructions must provide specific analysis guidelines relevant to the agent's role
- Output Format must match the table-based refinement perspective specification
- Context Rules must explicitly list which context files to load

**Extensibility Requirement (NFR4):**
- Adding a 4th agent requires only creating a new Markdown file in `agents/`
- No code changes should be needed in commands or workflows
- The new agent's `name` field can be added to config.yaml's `active_agents` array
- This enables the plugin architecture pattern from the architecture document
