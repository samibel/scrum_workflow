# Ticket Creation Workflow

Step-by-step workflow for the `/scrum-create-ticket` command. Transforms a natural language idea into a structured story file conforming to the story file schema (schema_version: 1). The workflow parses input, evaluates quality, loads project context, generates story content, estimates complexity, and writes the complete story file atomically.

## Prerequisites

- Framework installed with templates in `scrum_workflow/templates/`
- Project context generated via `/scrum-create-project-context` (context/index.md must exist) - optional but recommended
- Estimation reference data available at `scrum_workflow/data/estimation-reference.yaml`

## Step 0: Validation

Validate prerequisites and guard conditions before any processing begins.

### Step 0.1: Prerequisite Validation

Invoke `scrum_workflow/skills/prerequisite-validation/SKILL.md` to check for project context:

- Check if `context/index.md` exists at project root
- If context exists, it will be loaded in Step 3 to improve story quality
- If context does not exist, issue a warning but proceed (context is optional for `/scrum-create-ticket`)

**On missing context**, warn the user:

```
Warning: Project context not found. Story creation will proceed, but quality may be lower.
Tip: Run '/scrum-create-project-context' first to generate project context files for better story quality
```

**Note**: Unlike other commands, `/scrum-create-ticket` does NOT require project context to exist. It is optional but recommended.

### Step 0.2: Status Guard Validation

Invoke `scrum_workflow/skills/status-guard-validation/SKILL.md` to check if story file already exists:

- Check if `sprints/SW-XXX/story.md` already exists
- Guard condition: Story file must NOT exist for `/scrum-create-ticket`

**On guard condition failure** (story file already exists), halt with error:

```
Error: Story file 'sprints/SW-XXX/story.md' already exists
Fix: Delete the existing story file first, or use a different ticket number
```

### Step 0.3: Validation Halt

If any validation fails, halt the workflow immediately. Do not proceed to input parsing or any subsequent steps.

## Step 1: Input Parsing

Parse the user input to extract the ticket number and description.

**Expected input format:** `/scrum-create-ticket SW-XXX "description"`

### Step 1.1: Extract Ticket Number and Description

Extract the ticket number (e.g., `SW-103`) and the natural language description string from the user input.

### Step 1.1a: Validate Description Non-Empty

Validate that the extracted description is non-empty and contains at least one non-whitespace character.

**Validation rules:**

- Description must not be empty or contain only whitespace
- Minimum 1 non-whitespace character required

**On invalid description**, return an actionable error:

```
Error: Description cannot be empty. Please provide a description for the ticket.
Fix: Re-run the command with a description, e.g., /scrum-create-ticket SW-104 "your description here"
```

### Step 1.2: Validate Ticket Format

Validate that the ticket number matches the `SW-XXX` format where XXX is a zero-padded 3-digit number.

**Validation rules:**

- Prefix must be `SW-`
- Number must be exactly 3 digits with leading zeros (e.g., `001`, `042`, `103`)
- Valid range: `SW-001` through `SW-999`

**On invalid format**, return an actionable error:

```
Error: Invalid ticket format '<input>'. Expected format: SW-XXX (e.g., SW-001, SW-103)
Fix: Provide a ticket number in the format SW-XXX where XXX is a zero-padded 3-digit number
```

## Step 2: Vagueness Check

Evaluate the quality of the user's description to determine whether guided mode is needed.

### Step 2.1: Invoke Guided Mode Skill

Invoke `scrum_workflow/skills/guided-mode/SKILL.md` to evaluate the input description against vagueness detection criteria:

- **Input length threshold**: Fewer than approximately 20 words triggers guided mode
- **Missing key elements**: Two or more of who/what/where/why are absent
- **Ambiguous terms**: Vague terms like "improve", "fix", "update", "better" without clarifying scope

### Step 2.2: Handle Vague Input

If the guided mode skill determines the input is too vague, enter an iterative conversation loop:

1. **Present clarifying questions**: Display the structured questions generated by `scrum_workflow/skills/guided-mode/SKILL.md`, grouped by category (Scope, Technical, User Impact), to the user. Include a note that the user can type "/skip", "cancel", or "abort" at any time to exit guided mode and proceed with the current input
2. **Wait for user responses**: Collect the user's answers to the presented questions. Do not proceed until the user has responded
3. **Handle escape commands**: If the user types "/skip", "cancel", or "abort" (case-insensitive), exit the conversation loop immediately and proceed to Step 3 with the current enriched description (original input + any responses collected so far)
4. **Evaluate sufficiency**: After receiving responses, evaluate whether sufficient detail has been gathered by checking the guided mode exit conditions:
   - All four key elements (who/what/where/why) have been addressed
   - The user explicitly confirms the input is complete (e.g., "that's all", "proceed", "create it")
   - Sufficient detail exists to produce a well-defined story with testable acceptance criteria
5. **Generate follow-up questions if needed**: If the exit conditions are NOT met, invoke `scrum_workflow/skills/guided-mode/SKILL.md` again with the combined context (original input + previous responses) to generate follow-up questions informed by what has already been answered. Return to step 1 of this loop
6. **Merge enriched description**: Once the exit condition is met, merge the original description with all collected responses into a single enriched description
7. Continue to Step 3 with the enriched description

### Step 2.3: Handle Detailed Input

If the input is sufficiently detailed (adequate length, all key elements present, no unscoped ambiguous terms), skip guided mode and proceed directly to Step 3 with the original description.

## Step 3: Context Loading

Load project context to inform story generation with project-specific knowledge.

### Step 3.1: Check for Project Context

Check if `context/index.md` exists at the project root.

**If not found**, return an actionable error:

```
Error: Project context not found. Run '/scrum-create-project-context' first
Fix: Run '/scrum-create-project-context' to generate project context files before creating tickets
```

### Step 3.2: Load Project Context Index

Read `context/index.md` to understand the project's structure, domains, technology stack, and agent loading map.

### Step 3.3: Load Relevant Domain Context

Based on keywords in the ticket description, identify and load relevant domain context files:

- API, endpoint, service, database keywords -- load `context/backend.md`
- UI, component, page, form keywords -- load `context/frontend.md`
- Test, coverage, assertion keywords -- load `context/testing.md`
- Deploy, CI, pipeline, Docker keywords -- load `context/devops.md`
- Architecture, pattern, design keywords -- load `context/architecture.md`

**Loading rules:**

- Load all domain files that match keywords in the description
- If a domain file is referenced but does not exist on disk, log a warning and continue with available context
- If no specific keywords match, load only `context/index.md`

**On missing domain file**, warn the user but proceed:

```
Warning: Domain context file 'context/{domain}.md' not found. Proceeding with available context.
Tip: Ensure all referenced domain files exist or run '/scrum-create-project-context' to regenerate context files.
```

## Step 4: Story Generation

Generate structured story content from the user input enriched with project context.

### Step 4.1: Generate Story Title

Derive a concise, descriptive title from the ticket description. The title should clearly communicate the feature or change in a few words.

### Step 4.2: Generate Story Description

Produce a structured description section that:

- Clearly states the purpose of the feature or change
- References relevant project components, services, or modules identified from context
- Describes the expected behavior and user impact

### Step 4.3: Generate Acceptance Criteria

Produce acceptance criteria in **Given/When/Then** format:

- Each criterion must be specific and testable
- Cover the primary functionality described in the ticket
- Cover key edge cases and error scenarios
- Reference project-specific components and patterns where applicable

### Step 4.4: Generate Subtasks

Break the story into implementation subtasks:

- Each subtask represents a discrete unit of work
- Subtasks are ordered logically for implementation
- Use checkbox format (`- [ ]`) for tracking completion

## Step 5: Estimation

Generate an initial story point estimate based on the ticket complexity.

### Step 5.1: Load Estimation Reference

Read `scrum_workflow/data/estimation-reference.yaml` to access the story point scale (Modified Fibonacci: 1, 2, 3, 5, 8, 13) and complexity factors.

### Step 5.2: Evaluate Complexity Factors

Assess the story against complexity factors from the estimation reference:

- **Unknowns**: Degree of technical uncertainty
- **Dependencies**: Number of external dependencies or integration points
- **Testing requirements**: Breadth and depth of testing needed
- **Data migration**: Need to transform or migrate existing data
- **Cross-cutting concerns**: Changes affecting multiple system layers
- **Security sensitivity**: Features handling sensitive data or authentication
- **Performance requirements**: Specific performance targets

### Step 5.3: Calculate Estimate

Select the appropriate story point value from the Modified Fibonacci scale based on the complexity assessment. Follow estimation guidelines:

- Estimate relative to other stories in the project
- Include time for testing, code review, and documentation
- When uncertain, estimate higher rather than lower
- Flag stories estimated at 13 points for potential splitting

## Step 6: Sprint Folder Creation

Create the sprint folder structure for the new ticket.

### Step 6.1: Check for Existing Story

Check if `sprints/SW-XXX/story.md` already exists.

**If the file already exists**, return an actionable error:

```
Error: Story file 'sprints/SW-XXX/story.md' already exists. To re-create, delete the existing file first
Fix: Delete the existing story file or choose a different ticket number
```

### Step 6.2: Create Sprint Folder

Create the `sprints/SW-XXX/` directory if it does not already exist.

## Step 7: Story File Creation

Fill the story template with generated content and write the complete file atomically.

### Step 7.1: Load Story Template

Read the story template from `scrum_workflow/templates/story.md`.

### Step 7.2: Fill Template

Replace template placeholders with generated content:

**YAML frontmatter fields** (in this exact order):

| Field | Value |
|---|---|
| `schema_version` | `1` |
| `ticket` | `"SW-XXX"` (from input) |
| `title` | `"<generated title>"` (from Step 4.1) |
| `status` | `draft` |
| `estimation` | `<calculated>` (from Step 5.3) |
| `created` | `<today>` (ISO 8601 format: YYYY-MM-DD) |
| `updated` | `<today>` (ISO 8601 format: YYYY-MM-DD) |

**Markdown body sections:**

- **Description**: Generated story description (from Step 4.2)
- **Acceptance Criteria**: Generated Given/When/Then criteria (from Step 4.3)
- **Subtasks**: Generated implementation subtasks (from Step 4.4)

### Step 7.3: Write Complete File

Write the complete story file to `sprints/SW-XXX/story.md` in a **single write operation**.

**Critical (NFR1):** The entire file content -- YAML frontmatter and Markdown body -- must be written atomically. No partial writes that could leave the file in an inconsistent state or corrupt the frontmatter.

## Step 8: Confirmation

Display the created story summary to the user.

### Step 8.1: Display Summary

Present a confirmation message with:

- Ticket number and generated title
- File path: `sprints/SW-XXX/story.md`
- Status: `draft`
- Estimation: story points assigned
- Number of acceptance criteria generated
- Number of subtasks generated

### Step 8.2: Next Steps Guidance

Suggest next steps to the user:

- Review the generated story file and adjust as needed
- Run `/scrum-refine-ticket SW-XXX` to enter multi-agent refinement
- The story is now in `draft` status and can proceed through the state machine

## Write Boundaries

This workflow may write:

- `sprints/SW-XXX/story.md` -- The generated story file

This workflow may NOT write:

- `refinement.md` -- Managed by `/scrum-refine-ticket`
- `plan.md` -- Managed by readiness check
- `review-*.md` -- Managed by `/scrum-dev-story`
- `approval.md` -- Managed by approval workflow
- `scrum_workflow/` -- Framework files are read-only during execution
- `context/` -- Context files are managed by `/scrum-create-project-context`

## Validation Rules

- Generated `story.md` must have valid YAML frontmatter with all required fields
- YAML frontmatter field order must match: `schema_version`, `ticket`, `title`, `status`, `estimation`, `created`, `updated`
- Status must be set to `draft` (entry point to state machine)
- All YAML fields must use snake_case
- All file names and paths must use kebab-case
- Acceptance criteria must use Given/When/Then format
- File must be readable as standard Markdown without any tool (NFR8)
