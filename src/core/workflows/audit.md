# Audit Trail Workflow

**Purpose:** Central audit trail per story that records all transitions, agent actions, and artifact creation events for complete lifecycle traceability from draft to done.

**FR-38:** Central audit trail per story in `_scrum-output/audit/` with all transitions, agent actions, and artifact creation events.

**NFR-9:** Audit trail must be human-readable and Git-versionable (Markdown format).

---

## Audit Trail Entry Schema

```yaml
entries:
  - timestamp: 2026-04-09T12:00:00Z        # ISO 8601 UTC
    event_type: transition | action | artifact
    actor: human | architect-agent | dev-agent | system | refine-ticket | verify-skill
    details: "description of what happened"
    source_artifact: "refinement.md"         # optional, for artifact events
```

### Event Types

| Event Type | Source | Description |
|------------|--------|-------------|
| `transition` | status_history integration | Status changes (draft → refined → ...) |
| `action` | skill/agent executions | Manual edits, agent spawns, skill runs |
| `artifact` | file creation hooks | refinement.md, plan.md, verification-report.md, etc. |

### Valid Actor Values

| Actor | Description |
|-------|-------------|
| `human` | Manual human action |
| `architect-agent` | Architect agent execution |
| `dev-agent` | Developer agent execution |
| `system` | Automated system action |
| `{command}-skill` | Skill execution (e.g., refine-ticket, verify-skill) |

## Workflow: Recording Audit Entries

### Trigger Points

1. **Status Transitions:** Hook into status_history updates
   - When story status changes, append transition event
   - Copy from/to status, timestamp, trigger, actor from status_history entry

2. **Artifact Creation:** Hook into command workflows
   - When refinement.md is created → record artifact event
   - When plan.md is created → record artifact event
   - When verification-report.md is created → record artifact event
   - When review-N.md is created → record artifact event
   - When approval-N.md is created → record artifact event

3. **Agent Actions:** Record significant agent/skill events
   - When refinement agents spawn/complete
   - When review runs
   - When verification executes

### Integration with status_history (FR-7 / SC-7)

The status_history already contains transition data. The audit trail integration:

1. **Copies** transition events from status_history to audit trail
2. **Augments** with audit-specific fields (event_type, details)
3. **Maintains** chronological order

### Append-Only Rules

- **NEVER** modify existing entries
- **NEVER** delete entries
- **ONLY** append new entries at the end
- Use atomic write pattern (temp file + rename) to prevent corruption

### File Locking for Concurrent Writes

When multiple processes might write simultaneously:

1. Acquire exclusive lock before write
2. Append entry to file
3. Release lock

If lock cannot be acquired, retry with exponential backoff (max 3 attempts).

## Workflow: Querying Audit Trail

### Query Function Signature

```javascript
function getAuditTrail(ticketId, options = {}) {
  // options: { eventType, actor, startDate, endDate }
  // Returns: { entries: [...], summary: {...}, totalCount: number }
}
```

### Date Range Filtering

Audit trail supports filtering by date range to query events within a specific time window:

```javascript
// Filter by date range
getAuditTrail(ticketId, {
  startDate: "2026-04-01T00:00:00Z",
  endDate: "2026-04-30T23:59:59Z"
});

// Combined with event type and actor filters
getAuditTrail(ticketId, {
  eventType: "transition",
  actor: "human",
  startDate: "2026-04-01T00:00:00Z",
  endDate: "2026-04-30T23:59:59Z"
});
```

**Implementation notes:**
- Dates must be in ISO 8601 UTC format
- `startDate` is inclusive, `endDate` is inclusive
- Timestamp filtering happens at query time, not write time

## Complete Lifecycle Traceability (SC-7)

The audit trail, combined with status_history, provides complete traceability:

### Lifecycle Events Tracked

| Lifecycle Phase | Event Type | Source |
|-----------------|------------|--------|
| Story created | action | create-ticket |
| Status: draft → refined | transition | status_history |
| refinement.md created | artifact | refine-ticket |
| Status: refined → ready-for-dev | transition | status_history |
| plan.md created | artifact | refine-story |
| Status: ready-for-dev → in-progress | transition | status_history |
| Status: in-progress → review | transition | status_history |
| verification-report.md created | artifact | verify |
| review-N.md created | artifact | review-story |
| Status: review → approved | transition | status_history |
| approval-N.md created | artifact | approve |
| Status: approved → done | transition | status_history |

### Human-Readable Output

All audit entries use:
- ISO 8601 UTC timestamps (machine-parseable, human-readable)
- Markdown table format for structured data
- Descriptive details fields
- Clear event type labels

### Git-Versionable

- Markdown format (text-based, diff-friendly)
- YAML frontmatter for machine parsing
- Chronological order preserved in file

## Error Handling

### File Write Errors

1. **Permission denied:** HALT with clear error message
2. **Disk full:** HALT with clear error message
3. **Concurrent write conflict:** Retry with exponential backoff (max 3)
4. **Corrupt file:** Do NOT attempt recovery; HALT and report

### Query Errors

1. **File not found:** Return empty entries array, not error
2. **Parse error:** Return error with file location for manual review
3. **Invalid ticket ID:** Return validation error before file access
